## Overview

The pixi-build backend protocol uses JSON-RPC 2.0 for communication between the frontend (pixi) and backend implementations. Backends can communicate over:

- **stdin/stdout** - Default mode for subprocess-based backends
- **HTTP** - Alternative mode for network-based communication on `127.0.0.1`

The protocol is defined in `/home/daytona/workspace/source/crates/pixi-build-backend/src/protocol.rs` and `/home/daytona/workspace/source/crates/pixi-build-backend/src/server.rs`.

## Protocol Lifecycle

### 1. Negotiate Capabilities

**Method:** `pixi/negotiate_capabilities`

The first RPC call to establish what features the backend supports.

<ResponseField name="params" type="NegotiateCapabilitiesParams">
  Parameters for capability negotiation
</ResponseField>

<ResponseField name="result" type="NegotiateCapabilitiesResult">
  <Expandable title="properties">
    <ResponseField name="capabilities" type="BackendCapabilities">
      Capabilities supported by the backend
      <Expandable title="properties">
        <ResponseField name="providesCondaMetadata" type="boolean" default="false">
          Whether the backend can provide metadata about source packages
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

### 2. Initialize

**Method:** `pixi/initialize`

Initializes the backend with source directory and project information.

<ResponseField name="params" type="InitializeParams">
  <Expandable title="properties">
    <ResponseField name="sourceDir" type="string" required>
      Absolute path to the directory containing source files
    </ResponseField>
    <ResponseField name="capabilities" type="FrontendCapabilities" required>
      Capabilities provided by the frontend
    </ResponseField>
    <ResponseField name="project_model" type="ProjectModel">
      Optional project model passed to the backend. See [ProjectModel](/api/project-model) for details.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="result" type="InitializeResult">
  <Expandable title="properties">
    <ResponseField name="capabilities" type="BackendCapabilities" required>
      Backend capabilities (see Negotiate Capabilities)
    </ResponseField>
  </Expandable>
</ResponseField>

### 3. Conda Outputs

**Method:** `pixi/conda_outputs`

Queries the backend for available package outputs and their metadata.

<ResponseField name="params" type="CondaOutputsParams">
  <Expandable title="properties">
    <ResponseField name="work_directory" type="string" required>

    </ResponseField>
    <ResponseField name="channels" type="array">

    </ResponseField>
    <ResponseField name="variant_files" type="array">
      Paths to variant configuration files
    </ResponseField>
    <ResponseField name="variant_configuration" type="object">
      Variant configuration for build matrices

  </Expandable>
</ResponseField>

<ResponseField name="result" type="CondaOutputsResult">
  <Expandable title="properties">
    <ResponseField name="outputs" type="array" required>
      Array of CondaOutput objects describing available packages

        <ResponseField name="name" type="string" required>
          Package name

        <ResponseField name="version" type="string">
          Package version
        </ResponseField>
        <ResponseField name="build" type="string">
          Build string
        </ResponseField>
        <ResponseField name="subdir" type="string" required>
          Subdirectory/platform (e.g., "linux-64")
        </ResponseField>
        <ResponseField name="variant" type="object">
          Variant configuration used for this output
        </ResponseField>
        <ResponseField name="metadata" type="CondaOutputMetadata">
          Package metadata including dependencies and run exports

    <ResponseField name="source_globs" type="array">

    </ResponseField>
  </Expandable>

## Debug Output

The server automatically logs debug information to the work directory:

- `{work_directory}/debug/project_model.json` - Project model received during initialization
- `{work_directory}/debug/conda_outputs_params.json` - Parameters for conda_outputs call
- `{work_directory}/debug/conda_outputs_response.json` - Response from conda_outputs
- `{work_directory}/debug/conda_outputs_error.json` - Error if conda_outputs failed
- `{work_directory}/debug/conda_build_v1_params.json` - Parameters for conda_build_v1 call
- `{work_directory}/debug/conda_build_v1_response.json` - Response from conda_build_v1
- `{work_directory}/debug/conda_build_v1_error.json` - Error if conda_build_v1 failed

## Error Handling

Errors are returned using JSON-RPC error responses with code `-32000` (Server Error). The error data field contains a detailed diagnostic report in JSON format generated by miette.

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": -32000,
    "message": "Error description",
    "data": {
      // Miette diagnostic information
    }
  },
  "id": 1
}
```

## Implementation Guide

To implement a pixi-build backend:

1. Implement the `ProtocolInstantiator` trait:
   - `negotiate_capabilities()` - Return backend capabilities
   - `initialize()` - Set up backend state and return a Protocol instance

2. Implement the `Protocol` trait:
   - `conda_outputs()` - Return available package outputs
   - `conda_build_v1()` - Build a specific package output

3. Create a server:
   ```rust
   let server = Server::new(your_instantiator);
   server.run().await?; // stdin/stdout mode
   // or
   server.run_over_http(port)?; // HTTP mode
   ```

See the [Python Backend](/backends/pixi-build-python) and [Rust Backend](/backends/pixi-build-rust) for complete implementation examples.
